====================================================================
1) RINGKASAN SISTEM
====================================================================
Aplikasi ini adalah sistem manajemen restoran/kafe full-stack dengan modul POS,
inventory, kitchen display, purchasing, HR (attendance/payroll), laporan, member,
dan QR order.

Komponen utama:
- Frontend: React + Vite (resto-frontend)
- Backend API: PHP (plain MVC-style) + PDO + JWT (resto-backend/api)
- Database: MySQL/MariaDB (nama DB saat ini: restotaskora)
- Payment gateway: Midtrans (snap/webhook)
- Push/email: web-push + PHPMailer
- Deployment sekarang: subdomain cafe.quantum.or.id via Cloudflare Tunnel

Catatan penting:
- Dokumentasi schema DB di bawah disusun dari pembacaan source code karena file
  dump SQL tidak ditemukan di repository.

====================================================================
5) ALUR PROGRAM (END-TO-END)
====================================================================
5.1 Alur startup frontend
1. Browser load index.html
2. main.jsx render App dengan AppErrorBoundary
3. App.jsx menjalankan checkLicenseAndSettings()
4. App fetch GET public/info untuk nama/logo outlet
5. Router menentukan halaman berdasarkan path
6. Halaman protected cek token di localStorage

5.2 Alur request backend
1. Semua request masuk ke api/index.php?route=...
2. Router switch-case memetakan route + HTTP method ke controller
3. Controller authenticate (jika butuh) via JWT token
4. Controller memanggil model
5. Model eksekusi query SQL via PDO
6. Response JSON dikembalikan ke frontend

5.3 Alur transaksi POS
1. Kasir checkout dari halaman POS
2. Backend TransactionController::checkout()
3. Model Transaction::create() simpan header transaksi + item
4. Stock bahan berkurang berdasarkan recipe (product_recipes/variant_recipes)
5. Log mutasi stok dicatat ke stock_logs
6. Jika member: poin dihitung dan diupdate (point_logs)
7. Jika pending/payment method tertentu, status disesuaikan

5.4 Alur void/split/pay pending
- void: status transaksi + item diubah, stok dikembalikan, table dibebaskan
- split: sebagian item dipindah ke transaksi baru, total direkalkulasi
- pay pending: menyelesaikan transaksi yang sebelumnya belum lunas

5.5 Alur QR order (public)
1. Customer akses menu publik via route public/menu
2. Customer submit order ke public/order
3. Kitchen menerima item pending via kitchen/orders
4. Status item diperbarui via kitchen/update-status

10) KELEMAHAN SAAT INI DAN USULAN PERBAIKAN
====================================================================
10.1 Kelemahan arsitektur
1. Router API masih switch-case besar (index.php sangat panjang).
   Dampak: maintainability rendah, rawan konflik saat tim membesar.
   Usulan:
   - Refactor ke router map terpisah per module (auth.php, pos.php, inventory.php).
   - Tambahkan middleware chain (auth, role, validation).

2. Belum ada versi API (v1/v2).
   Dampak: breaking change sulit dikontrol.
   Usulan:
   - Gunakan prefix route versioning.

3. Banyak logic bisnis berada di model transaction yang sangat gemuk.
   Dampak: sulit testing, sulit debug.
   Usulan:
   - Pecah ke service: TransactionService, StockService, PointsService.

10.2 Kelemahan kualitas data/logika
1. Query SQL tidak selalu pakai transaksi database di semua alur mutasi multi-step.
   Dampak: data inkonsisten saat error di tengah proses.
   Usulan:
   - Standarkan beginTransaction/commit/rollback di semua operasi kritikal.

2. Potensi race condition stok saat checkout bersamaan.
   Dampak: stok bisa negatif/oversell.
   Usulan:
   - Lock baris ingredient (SELECT ... FOR UPDATE) saat deduct stock.
   - Tambahkan validasi stok minimum sebelum commit.

3. Beberapa fallback config masih mengarah ke brand/domain lama pada helper tertentu.
   Dampak: inkonsistensi branding/security metadata.
   Usulan:
   - Audit fallback env di seluruh helper dan satukan naming.

10.3 Kelemahan kinerja
1. Bundle frontend cukup besar (chunk besar >500KB).
   Dampak: load lambat di jaringan mobile.
   Usulan:
   - Lanjutkan code-splitting granular per page/feature.
   - Dynamic import library berat (face-api/recharts) hanya saat dipakai.

2. Penggunaan localStorage cukup banyak untuk settings/session parsial.
   Dampak: potensi stale cache dan perilaku tak sinkron antar tab.
   Usulan:
   - Simpan sumber kebenaran di backend, localStorage hanya cache ringan.
   - Tambah invalidation strategy.

10.4 Kelemahan keamanan
1. Belum terlihat rate limiting endpoint login/forgot password.
   Dampak: rentan brute force/abuse OTP.
   Usulan:
   - Tambahkan rate limiter per IP + per identifier.

2. Secret dan credential tampak masih placeholder/dev-style di env.
   Dampak: risiko keamanan production.
   Usulan:
   - Rotate JWT secret, gunakan secret manager, harden env policy.

3. Belum ada audit log terstruktur untuk aktivitas sensitif.
   Dampak: sulit forensik.
   Usulan:
   - Buat tabel audit_logs untuk login, reset password, void, perubahan setting.

10.5 Kelemahan observability/testing
1. Belum ada unit test/integration test otomatis di repo.
   Dampak: regresi mudah lolos.
   Usulan:
   - Tambah test backend (PHPUnit/Pest) untuk auth, POS, inventory.
   - Tambah frontend test untuk route kritikal (login, checkout).

2. Logging error belum terstruktur sentral.
   Dampak: troubleshooting lambat.
   Usulan:
   - Gunakan format JSON logs + request ID.
   - Tambahkan health endpoint dan dashboard monitoring.
